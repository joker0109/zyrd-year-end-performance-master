## 2. 产品概述

### 2.1 产品背景
公司每年需要对全体员工进行年度绩效评分，评分采用多层级互评机制，确保评价结果的公平性和全面性。

### 2.2 产品目标
- 实现员工分级评分流程的线上化管理
- 自动计算并生成员工最终绩效评级
- 提供评分数据的可视化展示和导出功能

## 4. 评分规则详解

### 4.1 评分标准

| 评分等级 | 代码 | 分值区间 | 说明 |
|----------|------|----------|------|
| A - 优秀 | `GRADE_A` | >95分 | 表现卓越 |
| B - 良好 | `GRADE_B` | 70-94分 | 表现良好 |
| C - 合格 | `GRADE_C` | 50-69分 | 达到要求 |
| D - 不合格 | `GRADE_D` | <50分 | 未达标准 |

### 4.2 各层级评分权重规则

#### 4.2.1 经理层评分权重

| 评分来源 | 权重 | 说明 |
|----------|------|------|
| 董事长评分 | 40% | 仅董事长对经理层的评分 |
| 全体员工评分 | 60% | 经理层互评 + 中层领导评分 + 普通员工评分 |

#### 4.2.2 中层领导评分权重

| 评分来源 | 权重 | 说明 |
|----------|------|------|
| 领导评分 | 40% | 董事长评分 + 经理层评分 |
| 全体员工评分 | 60% | 中层互评 + 普通员工评分 |

#### 4.2.3 普通员工评分权重

| 评分来源 | 权重 | 说明 |
|----------|------|------|
| 领导评分 | 40% | 董事长评分 + 经理层评分 + 中层评分 |
| 全体员工评分 | 60% | 普通员工互评 |

### 4.3 有效票数限制规则

> **重要约束1**：每人只能提交一次投票，提交后不可再次提交
> 
> **重要约束2**：投票人投出的A级票有数量上限限制

| 被评人层级 | 优秀票上限 | 说明 |
|------------|-----------|------|
| 经理层 | 1票 | 每人最多给1位经理层投优秀 |
| 中层领导 | 3票 | 每人最多给3位中层投优秀 |
| 普通员工 | 5票 | 每人最多给5位普通员工投优秀 |

**强制规则**：
1. 所有被展示的人员必须评分，不允许留空
2. **每人仅限提交一次投票，提交后不可再次修改**
3. **所有人不能给自己投票**

### 4.4 投票选项显示

投票选项使用中文显示：

| 代码值 | 显示文本 | 颜色标识 |
|--------|----------|----------|
| A | 优秀 | 绿色 |
| B | 良好 | 蓝色 |
| C | 合格 | 橙色 |
| D | 不合格 | 红色 |

### 4.5 防自投规则

投票时，系统自动过滤当前登录用户，不允许给自己投票：

```javascript
function getVoteTargets(voterId, voterLevel) {
    return employees.filter(e => {
        if (e.level === 1) return false; // 董事长不参与评分
        return e.id !== voterId; // 不能给自己投票
    });
}
```

## 5. 评分计算逻辑

### 5.1 数据定义

```
员工评分数据结构：
{
  employeeId: string,           // 员工唯一标识
  level: number,                // 员工层级 1/2/3/4
  leaderVotes: {                // 领导评分票数统计
    A: number,                  // 领导评A的票数
    B: number,                  // 领导评B的票数
    C: number,                  // 领导评C的票数
    D: number                   // 领导评D的票数
  },
  allStaffVotes: {              // 全体员工评分票数统计（不含董事长）
    A: number,
    B: number,
    C: number,
    D: number
  },
  leaderTotalVotes: number,     // 领导评分总票数
  allStaffTotalVotes: number    // 全体员工评分总票数
}
```

### 5.2 评分计算步骤

#### 步骤1：确定评分来源范围

| 被评人层级 | 领导范围（40%权重） | 全体员工范围（60%权重） |
|------------|---------------------|-------------------------|
| 经理层 | 董事长 | 经理层 + 中层 + 普通员工 |
| 中层 | 董事长 + 经理层 | 中层 + 普通员工 |
| 普通员工 | 董事长 + 经理层 + 中层 | 普通员工 |

#### 步骤2：统计各等级票数

```
对于每个被评员工：

领导评分票数统计：
  - leaderVotes.A = 领导评A的票数
  - leaderVotes.B = 领导评B的票数
  - leaderVotes.C = 领导评C的票数
  - leaderVotes.D = 领导评D的票数
  - leaderTotalVotes = leaderVotes.A + leaderVotes.B + leaderVotes.C + leaderVotes.D

全体员工评分票数统计（不含董事长）：
  - allStaffVotes.A = 全体员工评A的票数
  - allStaffVotes.B = 全体员工评B的票数
  - allStaffVotes.C = 全体员工评C的票数
  - allStaffVotes.D = 全体员工评D的票数
  - allStaffTotalVotes = allStaffVotes.A + allStaffVotes.B + allStaffVotes.C + allStaffVotes.D
```

#### 步骤3：计算各等级得分

```
对于每个评分等级 G ∈ {A, B, C, D}：

等级G得分 = [
  (leaderVotes.G / leaderTotalVotes) * 0.4 +
  (allStaffVotes.G / allStaffTotalVotes) * 0.6
] * 100

注意：
- 如果 leaderTotalVotes = 0，则领导评分部分为0
- 如果 allStaffTotalVotes = 0，则全体员工评分部分为0
```

#### 步骤4：计算总分

```
总分数 = A等级得分 + B等级得分
```

#### 步骤5：确定最终评级

| 总分区间 | 最终评级 |
|----------|----------|
| 总分 > 95 | 优秀（A） |
| 70 ≤ 总分 ≤ 94 | 良好（B） |
| 50 ≤ 总分 < 70 | 合格（C） |
| 总分 < 50 | 不合格（D） |

### 5.3 计算示例

**示例：某经理层员工评分计算**

```
假设数据：
- 董事长给该经理评分：A
- 其他投票情况：
  - 领导评分（仅董事长）：A=1, B=0, C=0, D=0，领导总票数=1
  - 全体员工评分：A=3, B=5, C=2, D=0，员工总票数=10

计算过程：

A等级得分 = [(1/1)*0.4 + (3/10)*0.6] * 100 = [0.4 + 0.18] * 100 = 58
B等级得分 = [(0/1)*0.4 + (5/10)*0.6] * 100 = [0 + 0.3] * 100 = 30
C等级得分 = [(0/1)*0.4 + (2/10)*0.6] * 100 = [0 + 0.12] * 100 = 12
D等级得分 = [(0/1)*0.4 + (0/10)*0.6] * 100 = 0

总分数 = 58 + 30 = 88分

最终评级：88分 ∈ [70, 94] → 良好（B）
```

#### 6.6.2 登录与会话管理

**登录方式：手机号 + 密码**

**登录流程：**
```
1. 用户输入手机号和密码
2. 后端验证手机号存在且密码匹配（MD5校验）
3. 生成 JWT Token（有效期2小时），存入 Redis（key: perf:login:{token}，TTL: 7200s）
4. 前端将 Token 存入 localStorage
5. 后续所有请求在 HTTP Header 携带：Authorization: Bearer {token}
6. 后端拦截器双重验证：JWT 签名 + Redis 存在性（支持主动踢出）
```

**会话恢复流程：**
```
1. 页面加载时读取 localStorage 中的 token
2. 携带 token 调用 GET /api/employee/me 验证是否有效
3. 有效：直接恢复登录状态，无需重新输入账号密码
4. 无效（过期/被踢出）：清除 localStorage，跳转登录页
```

**退出登录流程：**
```
1. 调用 POST /api/employee/logout
2. 后端删除 Redis 中对应的 Session（perf:login:{token}）
3. 前端清除 localStorage 中的 token
4. 重置页面状态，跳转到登录页
```

**员工登录信息字段：**

**会话配置：**

| 配置项 | 值 | 说明 |

- **行**：被投票人（所有参与评分的员工）
- **列**：投票人（所有参与投票的员工）
- **单元格**：投票等级（优秀/良好/合格/不合格）
- **最后两列**：得分和最终评级

**矩阵示例：**
```
┌──────────┬────────┬────────┬────────┬────────┬────────┐
│ 被投票人 │ 王经理 │ 李经理 │ 刘主管 │ 得分   │ 评级   │

│ 王经理   │   -    │ 优秀   │ 良好   │  88.5  │   B    │
│ 刘主管   │ 良好   │ 合格   │   -    │  75.2  │   B    │
│ 员工甲   │ 优秀   │ 优秀   │ 良好   │  92.0  │   B    │
```

### 7.2 核心表结构

#### employees（员工表）

```sql
CREATE TABLE employees (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    employee_id VARCHAR(50) NOT NULL UNIQUE COMMENT '员工编号',
    name VARCHAR(100) NOT NULL COMMENT '姓名',
    phone VARCHAR(20) NOT NULL UNIQUE COMMENT '手机号（登录凭证）',
    password VARCHAR(100) NOT NULL COMMENT '登录密码（MD5加密）',
    department VARCHAR(100) NOT NULL COMMENT '部门',
    level TINYINT NOT NULL COMMENT '层级：1董事长 2经理 3中层 4普通',
    position VARCHAR(100) COMMENT '职位',
    status TINYINT DEFAULT 1 COMMENT '状态：0离职 1在职',

    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_level (level),
    INDEX idx_department (department),
    INDEX idx_status (status)
) COMMENT='员工信息表';
```

#### votes（投票记录表）

```sql
CREATE TABLE votes (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    vote_session_id BIGINT NOT NULL COMMENT '投票批次ID',
    voter_id VARCHAR(50) NOT NULL COMMENT '投票人ID',
    voter_level TINYINT NOT NULL COMMENT '投票人层级',
    target_id VARCHAR(50) NOT NULL COMMENT '被评人ID',
    target_level TINYINT NOT NULL COMMENT '被评人层级',
    grade ENUM('A','B','C','D') NOT NULL COMMENT '评分等级',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_vote_session_voter_target (vote_session_id, voter_id, target_id),
    INDEX idx_voter (voter_id),
    INDEX idx_target (target_id),
    INDEX idx_vote_session (vote_session_id)
) COMMENT='投票记录表';

-- 每人每批次只能提交一次投票的约束
CREATE TABLE vote_submissions (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    vote_session_id BIGINT NOT NULL COMMENT '投票批次ID',
    voter_id VARCHAR(50) NOT NULL COMMENT '投票人ID',
    submit_time TIMESTAMP NOT NULL COMMENT '提交时间',

    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_session_voter (vote_session_id, voter_id),
    INDEX idx_voter (voter_id)
) COMMENT='投票提交记录表（用于控制每人只能提交一次）';
```

#### scores（评分结果表）

```sql
CREATE TABLE scores (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    level TINYINT NOT NULL COMMENT '员工层级',
    
    -- 领导评分票数
    leader_a_count INT DEFAULT 0 COMMENT '领导评A票数',
    leader_b_count INT DEFAULT 0 COMMENT '领导评B票数',
    leader_c_count INT DEFAULT 0 COMMENT '领导评C票数',
    leader_d_count INT DEFAULT 0 COMMENT '领导评D票数',
    leader_total_count INT DEFAULT 0 COMMENT '领导评分总票数',
    
    -- 全体员工评分票数
    staff_a_count INT DEFAULT 0 COMMENT '员工评A票数',
    staff_b_count INT DEFAULT 0 COMMENT '员工评B票数',
    staff_c_count INT DEFAULT 0 COMMENT '员工评C票数',
    staff_d_count INT DEFAULT 0 COMMENT '员工评D票数',
    staff_total_count INT DEFAULT 0 COMMENT '员工评分总票数',

    score_c DECIMAL(5,2) DEFAULT 0 COMMENT 'C等级得分',
```

## 8. 接口设计

### 8.1 接口清单

#### 员工认证接口（context-path: /api/employee）

| 接口 | 方法 | 路径 | 鉴权 | 说明 |
|------|------|------|------|------|
| 登录 | POST | /api/employee/login | 无需 Token | 手机号+密码，返回 JWT Token |
| 退出登录 | POST | /api/employee/logout | 需要 Token | 销毁 Redis Session |
| 获取当前用户信息 | GET | /api/employee/me | 需要 Token | 会话恢复，返回当前登录用户信息 |
| 修改密码 | POST | /api/employee/change-password | 需要 Token | 旧密码验证后修改 |

#### 评分投票接口（context-path: /api/vote）

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 获取待评人员列表 | GET | /api/vote/targets | 获取当前用户可参与评分的人员 |
| 获取我已提交的投票 | GET | /api/vote/my-votes | 查看本人已提交的评分内容（回显用） |
| 提交投票 | POST | /api/vote/submit | 提交本次全部评分，每人仅限一次 |
| 查询评分结果 | GET | /api/vote/result | 查询指定员工的评分结果，支持 `year` 参数 |
| 获取统计数据 | GET | /api/vote/statistics | 全员统计，支持 `year` 参数（韩雪专属） |
| 获取投票详情矩阵 | GET | /api/vote/detail | 投票矩阵，支持 `year` 参数（韩雪专属） |
| 获取历史绩效年份 | GET | /api/vote/years | 返回所有有投票记录的年份列表 |

> **鉴权机制**：所有接口（除 /api/employee/login）均需在请求头携带 `Authorization: Bearer {token}`，由后端 `AuthInterceptor` 统一验证 JWT 签名和 Redis Session 存在性。

## 10. 异常处理

### 10.1 异常情况清单

| 异常场景 | 处理方式 | 提示信息 |
|----------|----------|----------|
| 优秀票超出限制 | 前端拦截+后端校验 | "经理层优秀票最多投1张，您已超出限制" |
| 存在未评分人员 | 提交时校验 | "还有X人未评分，请完成所有评分" |
| 评分已截止 | 禁止提交 | "评分已截止，无法提交" |
| 重复提交 | 禁止操作 | "您已完成投票，无法再次修改" |
| 计算时无数据 | 标记异常 | "该员工暂无评分数据" |
| 除零错误 | 特殊处理 | 某维度无票时该维度得分为0 |

### 10.2 边界情况处理

```
1. 某员工没有被领导评分
   - leaderTotalVotes = 0
   - 领导评分部分 = 0
   - 仅计算全体员工评分部分 * 60%

2. 某员工没有被员工评分
   - allStaffTotalVotes = 0
   - 全体员工评分部分 = 0
   - 仅计算领导评分部分 * 40%

3. 某员工完全未被评分
   - 所有票数为0
   - 总分为0
   - 最终评级为D（不合格）
```
